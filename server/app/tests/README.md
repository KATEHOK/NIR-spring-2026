# Тесты

Комплект тестов для проверки корректности и производительности связки сервера и БД
аутентификации (FastAPI + PostgreSQL).

Тесты разделены на четыре группы:

- **Интеграционные тесты API** – проверяют каждый эндпоинт по отдельности
  (статус-коды, структуру ответов, время выполнения).
- **Нагрузочное тестирование (Locust)** – имитирует одновременную работу
  нескольких независимых пользователей и измеряет пропускную способность
  сервера под нагрузкой.
- **Комплексные сценарии** – эмулируют полные цепочки действий пользователя
  (регистрация → вход → обновление токенов → выход) с ожиданием истечения
  токенов и фиксацией времени каждого шага.
- **Unit‑бенчмарки криптографических операций** – сравнивают время выполнения
  синхронных вызовов (bcrypt, JWT, AES‑GCM) с выносом в отдельные потоки
  (`asyncio.to_thread`). Помогают обосновать, какие операции стоит переводить
  в потоки, а какие оставлять синхронными.

Все тесты, кроме unit‑бенчмарков, выполняют реальные HTTP‑запросы к работающему
серверу и сохраняют времена ответов в CSV‑файлы. Unit‑бенчмарки работают
напрямую с модулем `crypt.py` и сервер не требуют.

## Требования

- Сервер аутентификации запущен и доступен по `http://127.0.0.1:8000`
  (или `http://localhost:8000`) – **не требуется для unit‑бенчмарков**.
- База данных проинициализирована (выполнена команда
  `python src/init_db.py` из `server/app/`) – **не требуется для unit‑бенчмарков**.
- Для нагрузочного тестирования желательно прогреть сервер перед запуском
  (см. примечания).

## Установка зависимостей

Для каждого набора тестов используйте свой `requirements.txt`:

```bash
# Интеграционные тесты
pip install -r tests/integration/requirements.txt

# Нагрузочное тестирование
pip install -r tests/locust/requirements.txt

# Комплексные сценарии
pip install -r tests/complex/requirements.txt

# Unit‑бенчмарки
pip install -r tests/unit/requirements.txt
```

## Запуск тестов

### Интеграционные тесты API

```bash
cd server/app
pytest tests/integration/test_api.py -v
```

Результаты сохраняются в `tests/integration/results_api_integrations.csv`.

### Нагрузочное тестирование (Locust)

```bash
cd server/app
locust -f tests/locust/locustfile.py --host http://localhost:8000
```

Откройте веб-интерфейс Locust (по умолчанию `http://localhost:8089`),
задайте количество пользователей (например 100) и скорость нарастания (10
польз./с), запустите тест.

Для автоматического сбора метрик используйте headless‑режим:

```bash
cd server/app
locust -f tests/locust/locustfile.py --host http://localhost:8000 \
       --headless --users 200 --spawn-rate 2 --run-time 2m \
       --csv=tests/locust/results_locust  # базовое имя CSV-файлов
```

После завершения теста в папке `tests/locust/` появятся файлы
`results_locust_stats.csv`, `results_locust_failures.csv` и HTML‑отчёт.

### Комплексные сценарии

```bash
cd server/app
python tests/complex/test_complex.py
```

Сценарии выполняются параллельно (threading). Полный прогон занимает около
2-х минут (при временах жизни токенов: 1 мин для access и 2 мин для
refresh). Результаты записываются в `tests/complex/results_complex.csv`.

### Unit‑бенчмарки криптографических операций

```bash
cd server/app
python tests/unit/benchmark_crypto_operations.py
```

Скрипт выполняет для каждой из тестируемых функций (`Password.hash/verify`,
`JWT.encode/decode`, `SymmetricEncryption.encrypt/decrypt`) два прогона:
- синхронный (последовательные вызовы в основном потоке),
- асинхронный (вызовы через `asyncio.to_thread`, конкурентно в пуле потоков).

Количество вызовов на функцию задаётся параметром `call_count` в `main()`
(по умолчанию 100). После выполнения в папке `tests/unit/` создаётся
файл `results_benchmark_sync_vs_threaded.csv`

Пример настройки количества вызовов (измените `call_count` в файле):
```python
# в конце файла benchmark_crypto_operations.py
if __name__ == "__main__":
    asyncio.run(main(call_count=50))  # вместо 100
```

## Собранные метрики

### Интеграционные тесты

Файл `results_api_integrations.csv`:

| Столбец       | Описание                              |
|---------------|---------------------------------------|
| `test_name`   | Имя тестовой функции                  |
| `endpoint`    | URL эндпоинта                         |
| `status_code` | HTTP‑статус ответа                    |
| `elapsed_ms`  | Время ответа сервера, миллисекунды    |

### Нагрузочное тестирование

CSV‑файлы Locust содержат агрегированную статистику по каждому эндпоинту
(количество запросов, процент ошибок, медианное и среднее время ответа,
процентили, RPS). Полный список полей см. в документации Locust.

### Комплексные сценарии

Файл `results_complex.csv`:

| Столбец           | Описание |
|-------------------|----------|
| `scenario`        | Номер сценария (1–4) |
| `step`            | Название шага |
| `status`          | HTTP‑статус или `OK` для итоговой строки |
| `elapsed_ms`      | Чистое время выполнения запроса, мс |
| `sleep_ms`        | Время ожидания (искусственная пауза), мс |
| `total_elapsed_ms`| Сумма `elapsed_ms + sleep_ms` для данного шага  |

Для итоговых строк `TOTAL_SCENARIO_X`:
- `elapsed_ms` – суммарное время всех запросов сценария,
- `sleep_ms` – суммарное время всех ожиданий,
- `total_elapsed_ms` – полное время сценария (запросы + ожидания).

### Unit‑бенчмарки

Файл `results_benchmark_sync_vs_threaded.csv`:

| Столбец           | Описание                                      |
|-------------------|-----------------------------------------------|
| `experiment_id`   | Уникальный идентификатор прогона              |
| `async_mode`      | `False` – синхронный режим, `True` – через `to_thread` |
| `func_name`       | Имя функции (`hash`, `verify`, `encode`, …)   |
| `calls_count`     | Количество выполненных вызовов                |
| `elapsed_ms`      | Общее время выполнения, миллисекунды          |

## Примечания

- Интеграционные тесты используют синхронный HTTP‑клиент (`httpx.Client`)
  с keep‑alive, поэтому время ответа отражает реальную производительность
  сервера без накладных расходов на установку соединения.
- Успешный `login-accept` не автоматизирован (требует корректного OTP),
  проверяются только негативные сценарии.
- Перед нагрузочным тестированием рекомендуется прогреть сервер, выполнив
  20–30 запросов к `/auth/touch-db` (это инициализирует пул соединений с БД
  и сетевые структуры).
- Комплексные сценарии корректно работают только при совпадении констант
  `ACCESS_TOKEN_EXPIRE_MINUTES` и `REFRESH_TOKEN_EXPIRE_MINUTES` в файле
  `test_complex.py` с соответствующими переменными окружения сервера.
- Все CSV‑файлы перезаписываются при каждом запуске тестов. Сохраняйте
  нужные результаты перед повторным прогоном.
